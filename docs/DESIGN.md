# Jquard — Design Document

Jquard is a mountable Rails engine that brings the [Filament](https://filamentphp.com) (PHP/Laravel)
admin-panel experience to Ruby on Rails: a chainable, declarative resource DSL that generates a
modern, Filament-look admin UI on top of 100% Rails defaults (Hotwire, Propshaft, Tailwind-compiled CSS).

> Status: planning phase (July 2026). Nothing below is implemented yet unless marked otherwise.

---

## 1. Vision

- **Filament-familiar**: a Filament developer should be able to squint and read a Jquard resource.
  Same file structure, same fluent chaining, same component names — the Ruby way (snake_case,
  symbols, blocks instead of closures).
- **Rails-native**: Turbo + Stimulus instead of Livewire + Alpine, ActiveRecord validations as the
  source of truth, standard controllers underneath. No JS framework, no API layer, no node
  requirement in the host app.
- **Zero-friction install**: `bundle add jquard`, one generator, done. The gem ships its own
  precompiled CSS/JS; the host app needs no Tailwind setup.
- **Open source**, MIT, released on RubyGems.

### Positioning

ActiveAdmin is aging, Administrate is deliberately minimal, Trestle is quiet, Avo is partly
paid. "The Filament experience, free and OSS, on Rails defaults" is an empty niche. Filament's docs
double as the feature spec.

---

## 2. How Filament works (research summary, v4)

Filament is a server-driven UI framework: admin UIs are defined entirely in PHP as trees of
configuration objects. Stack: **Livewire** (stateful server-rendered components), **Alpine.js**,
**Tailwind** (compiled into semantic `fi-*` classes + CSS variables so users theme without
recompiling). Packages:

| Package | Purpose |
|---|---|
| Panels | Shell: sidebar nav, topbar, dashboard, auth, global search, dark mode, tenancy |
| Schemas | Base component-tree system (v4 unified forms + infolists on it) |
| Forms | ~25 field types + layout components (Section, Grid, Tabs, Wizard) |
| Tables | Columns, search, sort, filters, pagination, row/bulk actions, reordering |
| Actions | Buttons that open confirmation/form modals and run logic |
| Infolists | Read-only detail views (TextEntry, IconEntry, …) |
| Notifications | Toasts + database notification center |
| Widgets | Dashboard stats, charts, table widgets |

Everything hangs off the **Resource**. `make:filament-resource Customer --generate` introspects the
DB and emits a resource class (glue), page classes, a form schema class, and a table class.
The fluent API (`TextInput::make('email')->email()->required()`) plus closure utility injection by
parameter name (`fn ($record, $state) => …`) is the core of its ergonomics.

## 3. File structure — and what each file is for

Generated by `bin/rails g jquard:resource Post` (mirrors Filament v4 exactly):

```
app/jquard/resources/posts/
├── post_resource.rb        # THE GLUE — model, navigation, wiring
├── pages/
│   ├── list_posts.rb       # the /posts screen        → renders posts_table
│   ├── create_post.rb      # the /posts/create screen → renders post_form (new record)
│   └── edit_post.rb        # the /posts/:id/edit screen → renders post_form (existing record)
├── schemas/
│   └── post_form.rb        # WHAT the form contains (fields, layout) — shared by create + edit
└── tables/
    └── posts_table.rb      # WHAT the index table contains (columns, search, actions)
```

The mental model — three different jobs:

1. **Content definitions** (`schemas/`, `tables/`) describe **what** is displayed. They are
   *shared, reusable* definitions:
   - `post_form.rb` is used by **both** the Create and the Edit page (and later by modal actions
     and relation managers). That's why it lives in its own file instead of inside either page.
   - `posts_table.rb` is used by the List page (and later by relation managers and table widgets).
   - Why "schemas" and not "forms"? In Filament v4 the same component-tree mechanism ("schema")
     powers forms, infolists (read-only detail views), and custom pages. The folder is named after
     the mechanism. When the View page arrives (v0.2), `schemas/post_infolist.rb` will sit next to
     `post_form.rb`.

2. **Pages** (`pages/`) describe the **screens that host that content** — one class per routable
   screen. They hold everything that is *per-screen*, not per-content: page title, breadcrumbs,
   header buttons, redirect targets, lifecycle hooks. Freshly generated they are nearly empty;
   they exist as the place where customization accumulates:

   ```ruby
   class Jquard::Resources::Posts::Pages::CreatePost < Jquard::Resources::Pages::CreateRecord
     def mutate_form_data_before_create(data)
       data.merge(author_id: current_user.id)
     end

     def redirect_url_after_create = resource.url(:index)
   end
   ```

   In Filament, pages are full-page Livewire components. In Jquard they are thin config classes;
   a single engine controller dispatches through them (routing in §6).

3. **The resource** (`post_resource.rb`) is the **manifest** that ties it together: which model,
   how it appears in navigation, which pages exist at which routes, the base query scope. It
   contains no UI itself.

```
                       PostResource  (model, nav, routes)
                      /      |      \
        Pages::ListPosts  Pages::CreatePost  Pages::EditPost
                |                 \            /
        Tables::PostsTable      Schemas::PostForm
```

## 4. The DSL

### Resource

```ruby
class Jquard::Resources::Posts::PostResource < Jquard::Resource
  self.model = ::Post
  self.navigation_icon = "document-text"     # heroicons, same names as Filament
  self.navigation_group = "Blog"
  self.navigation_sort = 2
  self.record_title_attribute = :title

  def self.form(schema)   = Schemas::PostForm.configure(schema)
  def self.table(table)   = Tables::PostsTable.configure(table)

  def self.pages
    {
      index:  Pages::ListPosts.route("/"),
      create: Pages::CreatePost.route("/create"),
      edit:   Pages::EditPost.route("/:record/edit")
    }
  end

  # def self.record_query(scope) = scope.where(archived: false)   # ≙ getEloquentQuery()
end
```

There is exactly **one way** to define a resource: the split-file structure above. No inline
`form`/`table` shorthand (decided 2026-07-11 — one consistent way).

> **Implementation note (2026-07-14):** the compact class style shown in these samples
> (`class Jquard::Resources::Posts::PostResource`) cannot resolve relative sibling constants like
> `Tables::PostsTable` — Ruby's lexical constant lookup skips intermediate namespaces in compact
> declarations. Real files (and the generator) therefore use **nested module style**
> (`module Jquard / module Resources / module Posts / class PostResource`), which makes relative
> references work exactly like PHP's namespace imports. See
> `test/dummy/app/jquard/resources/posts/` for the canonical shape.

### Form schema

```ruby
class Jquard::Resources::Posts::Schemas::PostForm
  include Jquard::Schemas::Components   # brings TextInput, Select, … into constant scope

  def self.configure(schema)
    schema.components [
      Section.make("Content").columns(2).schema([
        TextInput.make(:title).required.max_length(255).column_span_full,
        Select.make(:status)
          .options(draft: "Draft", reviewing: "Reviewing", published: "Published")
          .default(:draft)
          .required,
        DateTimePicker.make(:published_at),
        Toggle.make(:featured),
        Textarea.make(:body).rows(10).column_span_full
      ])
    ]
  end
end
```

### Table

```ruby
class Jquard::Resources::Posts::Tables::PostsTable
  include Jquard::Tables::Components

  def self.configure(table)
    table
      .columns([
        TextColumn.make(:title).searchable.sortable,
        TextColumn.make(:status).badge.color(draft: :gray, published: :success),
        IconColumn.make(:featured).boolean,
        TextColumn.make(:published_at).date_time.sortable
      ])
      .record_actions([EditAction.make, DeleteAction.make])
      .default_sort(:created_at, :desc)
  end
end
```

### Blocks with Filament-style utility injection

Ruby can inspect block parameter *names* (`block.parameters`), so we clone Filament's closure
ergonomics — ask for what you need by naming it:

```ruby
TextColumn.make(:price).state { |record| record.cents / 100.0 }
DeleteAction.make.visible { |record, current_user| current_user.admin? && !record.locked? }
TextInput.make(:slug).default { |get| get[:title]&.parameterize }   # reactive — post-MVP
```

### Host installation

```bash
bundle add jquard
bin/rails generate jquard:install      # mounts engine at /admin, creates initializer
bin/rails generate jquard:resource Post
```

```ruby
# config/initializers/jquard.rb
Jquard.configure do |config|
  config.brand_name = "Acme"
  config.primary_color = :ruby         # default; any Tailwind palette name or custom hex
  config.authenticate_with { redirect_to main_app.root_path unless current_user&.admin? }
end
```

## 5. Stack translation

| Filament | Jquard | Notes |
|---|---|---|
| Livewire | Turbo Frames/Streams + plain controllers | Table sort/search/filter/pagination = GET params in a Turbo Frame. `live()` reactive fields are the hard part → roadmap v0.4. |
| Alpine.js | Stimulus | Dropdowns, modals, toggles. |
| Tailwind (`fi-*` classes) | Tailwind precompiled at gem-build time into `jq-*` classes + CSS variables | Host app needs no Tailwind/node. Propshaft serves one CSS file. Theming via CSS variables. |
| Blade component views | ViewComponent | See §8. |
| Eloquent | ActiveRecord | Generator introspection via `columns_hash`, `reflect_on_all_associations`, `validators`. |
| Laravel request validation | ActiveRecord model validations | See below. |

**Validation lives in the model; the DSL decorates it.** `.required`/`.max_length` render the
asterisk and HTML5 attributes; saving goes through normal AR validations; errors render
Filament-style under each field. The generator reads `Post.validators` to pre-chain `.required`.
Later, DSL rules can add form-object validation for fields with no model backing.

**Look & feel**: Filament v4, faithfully — layout, spacing, dark sidebar, toasts. But Jquard's
**default primary color is ruby red** (Ruby-language red, ~`#CC342D` / Tailwind rose family)
instead of Filament's amber, so a default install is visibly its own thing. Fully configurable.
Design tokens are dark-mode-ready from day one; the toggle ships in v0.2.

## 6. Engine architecture

- Standard mountable engine (`isolate_namespace Jquard`), mounted by the host:
  `mount Jquard::Engine => "/admin"`.
- **Host-side resource classes**: the engine registers `app/jquard` as a *namespaced* Zeitwerk
  root, so `app/jquard/resources/posts/post_resource.rb` defines
  `Jquard::Resources::Posts::PostResource` — mirroring `App\Filament\Resources\…`. (Same pattern
  Avo uses.)
- **Registry**: `Jquard::Resource.inherited` self-registers subclasses (slug → class). Rebuilt in
  `Rails.application.config.to_prepare` so dev-mode code reloading works.
- **Routing/dispatch**: engine routes use a dynamic segment
  (`":resource_slug" → Jquard::ResourcesController`); the controller resolves the slug via the
  registry, instantiates the matching Page class, builds the table/form config, applies
  search/sort/pagination to `resource.record_query(model.all)`, renders.
- **Request flow example**: `GET /admin/posts?sort=title` → `ResourcesController#index` →
  `PostResource` → `Pages::ListPosts` → `Tables::PostsTable.configure` → query + render. The table
  lives in a Turbo Frame; sort/search/paginate swap just the frame. Create/edit are plain form
  submissions; success emits a Turbo Stream toast.
- **No migrations** in MVP — Jquard is pure configuration over host models. (Database
  notifications will need a table later.)
- **View overriding**: hosts can override any engine view by shadowing its path — free
  customization escape hatch (applies to ERB; with ViewComponent, customization is DSL-driven +
  subclassing, which matches Filament's philosophy).
- **Zero runtime dependencies beyond Rails and ViewComponent**. Pagination is
  hand-rolled (~50 lines). Icons are inlined Heroicons SVGs. CSS/JS ship precompiled.

## 7. MVP (v0.1) requirements

1. **Shell**: Filament-look layout — dark sidebar with grouped nav + icons, topbar, breadcrumbs,
   brand name/color from config. Precompiled Tailwind, ruby-red default primary.
2. **Resource system**: `Jquard::Resource` base, `app/jquard` namespaced autoloading,
   self-registering reload-safe registry.
3. **List page**: `TextColumn` (+ `badge`, `date_time`, `limit`), `IconColumn` (`boolean`);
   global search over `searchable` columns; column sort; pagination with per-page selector;
   empty state; row actions (Edit, Delete with confirm modal); Create header action. All in
   Turbo Frames.
4. **Form + Create/Edit pages**: `TextInput` (`email`/`password`/`numeric` variants), `Textarea`,
   `Select`, `Checkbox`, `Toggle`, `DatePicker`, `DateTimePicker`, `Hidden`; `Section` + `Grid`
   layouts; chainables `label`, `placeholder`, `helper_text`, `required`, `default`, `disabled`,
   `column_span`; AR validation errors rendered Filament-style; toasts on save/delete.
5. **Generators**: `jquard:install`; `jquard:resource Post` introspecting `columns_hash`, Rails
   `enum`s and validators (string→TextInput, text→Textarea, boolean→Toggle/IconColumn,
   datetime→DateTimePicker, enum→Select, …).
6. **Quality**: Capybara system tests in `test/dummy` for the three flows; README; CI.

**Out of MVP scope**: authentication (delegated to host via `authenticate_with` hook only),
filters, bulk actions, relationships, View page, widgets, dark-mode toggle, reactivity.

## 8. View layer: ViewComponent (decided)

Every DSL config object (column, field, action, layout) renders through a
[ViewComponent](https://viewcomponent.org) — one class + template per component. A table page
renders one component per cell (25 rows × 6 columns = 150+ renders per page), which is where
ViewComponent's strengths pay off over ERB partials:

- explicit initializer interface instead of loosely passed locals;
- rendering logic as plain methods on the class instead of global helpers;
- unit-testable in isolation (render one component without a controller);
- ~10× faster than partial rendering in loops.

Used by Avo for exactly this job. Trade-off accepted: hosts cannot override component templates by
shadowing view paths the way they can with partials — customization is DSL options + subclassing,
matching Filament's philosophy. ViewComponent is Jquard's only runtime dependency besides Rails.

## 9. Roadmap

- **v0.2 — Table depth & viewing**: filters (Select, ternary, date-range), bulk actions,
  toggleable columns, View page + Infolist entries, ImageColumn, FileUpload (ActiveStorage),
  RichEditor (ActionText/Lexxy), dark-mode toggle.
- **v0.3 — Relationships**: `Select.relationship(:author, :name)` with search; multi-select /
  CheckboxList for HABTM; relation managers (embedded tables on Edit pages with
  create/attach/detach) + `jquard:relation-manager` generator; dot-notation columns
  (`author.name`) with auto eager-loading.
- **v0.4 — Actions & reactivity**: custom actions with forms in modals/slide-overs; `live()`
  fields via a Stimulus controller posting form state to a schema-refresh endpoint returning a
  Turbo Stream morph.
- **v0.5 — Panel maturity**: authorization (Pundit-convention policies), global search (Cmd+K),
  dashboard widgets (stats + charts), theming API, optional login module, multiple panels.
- **v1.0 — Release**: plugin/extension API, docs site, stable public API.
